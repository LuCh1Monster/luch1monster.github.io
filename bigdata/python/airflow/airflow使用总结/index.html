    本篇文档主要讲述 AirFlow 常规使用的模块以及一些参数。

    ¶1. Dag

    首先来看一个小例子，用来创建一个 DAG，但是未添加任何的 task 节点:

    from airflow import DAG
from datetime import datetime, timedelta

default_args = {'owner': 'admin',
                'depends_on_past': False,
                'start_date': datetime(2019, 1, 1, 8, 0),
                'email': 'xxxxxxx@qq.com',
                'email_on_failure': True,
                'email_on_retry': True,
                'retries': 1,
                'retry_delay': timedelta(minutes=1),
                'priority_weight': 10,
                'end_date': datetime(2020, 1, 1),
               }

dag = DAG(
    dag_id='test_dag',
    default_args=default_args,
    description='my first DAG',
	schedule_interval='30 8 * * *' 
	)

    ¶1.1 default_args

    首先来看 default_args，它是一个字典，包含了很多的一些常用参数，下面我们逐一介绍。

    ¶1) owner

    owner 表示该 DAG 的创建者人，一般都需要指定，并且这个 owner 不一定必须要是 AirFlow 的用户，你可以填写任意的字符串。

    ¶2) depends_on_past

    depends_on_past 表示该 DAG 是否依赖于上一次 DAG 执行状态。

    • 当 depends_on_past=True 时，表示当前 DAG 依赖于上一次运行的状态，如果上一次运行状态为 failed，则这一次该 DAG 不会运行。
    • 当 depends_on_past=False 时，表示当前 DAG 不依赖于上一次运行的状态，可以独立运行。

    ¶3) start_date

    start_date 表示任务开始的时间，理论上这个属性应该设置为任务部署的前一天，比如该任务要在 2019-01-02 的 8点半上线，则应该设置为 datetime(2019, 1, 1, 8, 30, 0)。至于为什么要设置为前一天，这是 AirFlow 内部机制决定的。

    当然，这个属性页也可以不用指定明确的日期时间，还可以设置成其他的一些日期值:

    • airflow.utils.dates.days_ago(1) : 表示前一天

    实际在部署时，还是使用确切的日期比较好，可以方便的看到这个任务在什么时间上线的。

    ¶4) end_date

    end_date 与 start_date 对应，表示该 DAG 的结束时间，一旦当前日期大于等于 end_date，则该任务不会继续自动调度。

    ¶5) catchup

    catchup 字面上的含义是追赶，具体的含义是，如果你的一个任务定义了 start_date，且这个 start_date 小于当前日期，比如 start_date 值为 2019-01-01，而当前日期为 2019-01-10，则一旦任务部署成功后，任务会自动会创建 dags 去执行 2019-01-02 ~ 2019-01-09 这段时间内任务。

    使用该参数需要格外注意两点:

    • 当任务需要补全历史数据时，这个参数是非常有用的。前提条件是，你的代码已经足够鲁棒性，因为在补全历史任务时，获取的数据不一定是当时的数据，有可能补全的数据都是今天的，这就可能造成数据冗余或者时间上浪费，正确的数据没有更新。
    • 实际部署时，还是建议大家将该参数设置为 False，通过手工的方式补数据更为稳妥。

    ¶6) email

    当任务失败时，设置失败邮件的接收者的邮箱，可以是一个邮箱，也可以是一堆人的邮箱。

    ¶7) email_on_failure

    email_on_failure 用来设置任务失败时是否发送邮件给指定接收人。

    ¶8) email_on_retry

    email_on_retry 用来设置任务 RETRY 时是否发送邮件给指定接收人。

    ¶9) on_failure_callback

    on_failure_callback 用来定义如果任务失败时，执行相应的动作，设置时为一个具体的 function。比如上述我们介绍了 email_on_failure 和 email_on_retry 的作用，但是真的任务失败了发送的是什么邮件内容呢，这就需要你定义一个 发送失败邮件的函数，一旦任务失败了就会执行该函数。当然，任务失败时你也可以定义其他的动作。相应的还有 on_success_callback 和 on_retry_callback 属性。

    ¶10) retries

    设置该 DAG 如果运行失败，重新尝试运行的最大次数。该属性的设置为全局的，即对于每一个 task，如果没有具体指定 task 的 retries 属性，则都会以 DAG 的 retries 作为 task 的重新尝试次数。

    ¶11) retry_delay

    retry_delay 用来设置失败重新运行的间隔时长，你可以设置为隔1分钟就重新执行一次，也可以设置为每10秒中重新执行一次，具体设置时可以参考以下参数:

    • timedelta(minutes=1) : 每间隔1分钟运行一次
    • timedelta(seconds=10) : 每间隔10秒运行一次

    ¶12) pool

    pool 用来设置进程池，它是一个分类限制并发量的设计，目前来说可以忽略，默认所有的Task都在一个 pool 里。如果该 DAG 指定了 pool 指定了明确的 Pool，那么如果其他任务也在执行该进程池，且 pool 的最大并行度为 1 时，则该 DAG 需要进行等待。

    ¶13) queue

    queue 表示该 DAG 属于哪个队列，默认是 default，决定实际执行任务会发送到哪个worker，只在 CeleryExcutor 部署方式中会用到。

    ¶14) priority_weight

    priority_weight 用来设置优先级权重，在任务需要排队时而你需要优先执行某些任务时会有用。

    ¶1.2 schedule_interval

    schedule_interval 用来指定任务的调度时间，可以使用 cron 时间调度格式，也可以使用 AirFlow 内置的调度格式，可以参考以下几种格式:

    • timedelta(days=1) : 每天进行调度，触发任务运行时间以 start_date 的时间为准。
    • @monthly: 每个月跑一次，其他选项还有 @once、@hourly、@daily、@weekly、@yearly 以及 None。None 表示不进行调度。同样地，触发任务运行时间以 start_date 的时间为准。
    • '30 8 * * *' : crontab 的时间格式，表示每天 8点半运行。

    ¶1.3 max_active_runs

    max_active_runs 用来设定该 DAG 下 Task 并行运行最大数量。

    ¶1.4 dagrun_timeout

    dagrun_timeout用来设置 DAG 的最大运行时间，可以设置为datetime.timedelta(minutes=60)，表示该任务运行时间最大为 60 分钟。

    ¶1.5 catchup

    该参数与 default_args 中的 catchup 类似。

    ¶2. PythonOperator

    ¶2.1 python_callable

    python_callable 用来指定该 TaskID 实际调用的函数。

    ¶2.2 on_failure_callback

    on_failure_callback 用来指定该 TaskID 失败后调用的函数。

    ¶2.1 provide_context

    provide_context 如果设置为 true，Airflow将传递一组可在函数中使用的关键字参数。 这组kwargs 完全对应于你在 jinja 模板中可以使用的内容。为此，您需要在函数头中定义 ** kwargs。

    ¶2.2 trigger_rule

    默认情况下，某个 task 是否执行，依赖于其父 task(直接上游任务)全部执行成功。airflow 允许创建更复杂的依赖。通过设置 operator 中的 trigger_rule 参数来控制:

    • all_success : 父 task 全成功

    • all_failed : 父task全failed或者upstream_failed状态

    • all_done : 父 task 全执行过，不管 success 或者 failed

    • one_failed : 当父task中有一个是failed状态时执行，不必等到所有的父task都执行

    • one_success : 当父task中有一个是success状态时执行，不必等到所有的父task都执行

    • dummy : 无条件执行

    该参数可以和 depends_on_past 结合使用，当设置为 true 时，如果上一次没有执行成功，这一次无论如何都不会执行。

    ¶2.3 op_args 和 ops_kwargs

    • op_args : 调用 callable 时将解压缩的位置参数列表
    • op_kwargs : 关键字参数字典，将在调用函数中解压

    ¶2.4 retries

    该参数与 default_args 的 retries 一样，用来设置 Task 失败后最大重新运行次数。

    ¶2.5 execution_timeout

    execution_timeout 表示执行此任务实例允许的最长时间，超过最长时间则任务失败。

    ¶2.7 简单示例

    from airflow.operators.python_operator import PythonOperator

def func_t1(check):
    if check:
        print("123")
    else:
        raise Exception("运行失败")

def func_t2(check):
    if check:
        print("123")
    else:
        print("456")

task_t1 = PythonOperator(
    task_id='task_t1',
    python_callable=func_t1,
    op_kwargs={'check': True},
    trigger_rule='all_failed',
    on_failure_callback=func_t2,
    retries=0,
    dag=dag
)

    ¶3. BaseSensorOperator

    ¶3.1 参数

    • soft_fail : 设置为 true 以将任务标记为失败时的 skipped

    • poke_interval : 作业在每次尝试之间应等待的时间(单位: 秒)

    • timeout : 超时时间

    ¶4. BranchPythonOperator

    DAG 中的任务可以选择分支，BranchPythonOperator 允许用户通过函数返回下一步要执行的 task的 id，从而根据条件选择执行的分支。azkaban 没有该功能。注意，BranchPythonOperator 下级task 是被 selected 或者 skipped 的分支。

    BranchPythonOperator 与 PythonOperator 非常相似，只是它需要一个返回 task_id 的python_callable。 返回 task_id，并跳过所有其他路径。Python 函数返回的 task_id 必须直接引用BranchPythonOperator 任务下游的任务。

    请注意，在 BranchPythonOperator 的下游任务中使用 depends_on_past=True，这在逻辑上是不合理的，因为 skipped 状态将总是因为他们过去的 successes 而造成 task 的堵塞。如果你想跳过一些任务，请记住你不能有一个空路径，如果是这样，那就做一个虚设任务。像这样，跳过虚拟任务“branch_false”。
