    1.1 数据架构师为啥选择Hadoop

    数据库的发展历史

    • 1970年，埃德加.弗兰克.科德，IBM公司研究员，发表题为“大型共享数据库的关系模型”的论文，被誉为“关系型数据库之父”
    • 1979年，因 IBM 战略主导分级结构数据业务并将萌芽期的关系型数据库结构业务删离；柏克莱 Ingres 数据库研究项目联合 Oracle 创建了第一个商业 RDBMS
    • 1988年，IBM 研究员率先提出并结识了数据仓库一词的行业标准
    • 2002年，Netezza(pronounced Ne-Tease-Ah)公司将存储、处理、数据库和分析融入到一个高性能数据仓库设备中，资源仓库软硬件整合数据库仓库数据库浮现
    • 2010年，IBM 以 17 亿美元收购 Netezza 并推出了 IBM Netezza 数据仓库一体机
    • 2008年，Facebook 基于静态赔处理的 Hadoop 之上封装并飞不了一个开源项目–数据仓库 Hive，可以将结构化的数据文件映射为一张数据表，并提供简单的 sql 查询功能，可以将 sql 语句转换为 MapReduce 任务进行运行
    • 2010年，Hbase 发行，在 Hadoop 之上提供了类似于 Bigtable 的能力，是一个适合于非结构化数据存储的数据库，采用基于列的而不是基于行的模式

    2.3 HDFS的安全设置及原理

    Hadoop自带两种安全机制: Simple 机制、Kerberos 机制

    • Simple 机制是 JAAS 协议与 delegation token 结合的一种机制，JAAS(Java Authentication and Authorization Service) java认证与授权服务

    • Kerberos 机制是基于认证服务器的一种方式

    HDFS安全机制

    Client 获取 namenode 初始访问认证(使用 Kerberos)后，后会获取一个 delegation token，这个 token 可以作为接下来访问 HDFS 或者提交作业的凭证。

    同样，为读取某个文件，client 首先要与 namenode 交互，获取对应 block 的 block access token，然后到相应的 dataname 上读取各个 block，而 datanode 在初始启动项 namenode 注册时，已经提前获取了这些 token，当 client要从 TaskTacker 上读取 block 时，首先验证 token，通过后允许读取。

    3. 趣谈Hadoop的计算资源设计

    3.1 计算资源概述

    计算资源(resource on the compution) 计算复杂性理论的一个术语。指计算中所需的各种资源一般地，各种计算模型的主要资源有并行时间、串行时间和空间三种

    • 并行时间和巡回

    并行时间一般指并行模型计算时所需部署，例如，向量机的自始至终执行指令的总条数。但对串行模型也可以定义一种称为巡回的资源。可以证明它相当于并行时间。对于多带图灵机，它是工作带头部改变方向的次数。一般地，巡回是周相的总数，而周相则是串行模型工作中的一个阶段。再次阶段中计算出来而记录在工作空间上的信息，不在被读到。

    • 串行时间

    计算过程中原始运算的总量。对于串行模型儿而言，它代表计算至始至终的总步数；对于并行模型而言，每一步可以同时作许多个原始的原酸，至始至终各步的原始运算数目的总和就是串行时间。

    • 空间

    在计算过程中需要记录下来以备后用的最大中间信息量

    相关名词 原理及优点 使用场景
    CPU(串行计算) Central Processing Unit 解释计算机指令以及处理计算机软件中的数据 基于单机或单服务器的计算
    并行计算 使用多种计算资源解决计算问题的过程 利用廉价的计算资源取大型计算机
    分布式并行计算 把一个需要巨大计算能力才能解决的问题分割成许多小的部分，然后把这些不法分配给多个计算机平行计算处理，把计算结果总和起来得到最终结果 可以利用分布式系统的优势和并行计算的方案来大幅降低硬件的成本

    5. Hive原理及应用场景

    5.1 Hive概述

    hive 是基于 Hadoop 的一个数据仓库工具，可以将结构化的数据文件映射为一张数据库表，并提供简单的 SQL 查询功能，可以家将 SQL 语句转换为 MapReduce 任务进行运行。

    其优点是学习成本低，可以通过类 SQL 语句快速实现简单的 MapReduce 统计，不必开发专门的 MapReduce 应用，十分适合数据仓库的统计分析。

    5.2 Hive的特点

    • 可扩展性: Hive 可以自由的扩展集群的规模，一般情况下不需要重启服务

    • 延展性: Hive支持用户自定义函数，用户可以根据自己的需求实现自己的函数

    • 容错性: 良好的容错性，节点出现问题SQL仍可完成执行

    5.3 Hive适用场景

    最佳适用场合:

    • 大数据的批处理作业，例如: 网络日志分析

    不适用的场景:

    • 不能在大规模数据集上实现低延迟快速的查询，例如: Hive 在几百 MB 的数据集上执行查询一般有分钟级的时间延迟
    • 不支持联机事物处理(OLTP)
    • Hive 不提供基于行级的数据更新操作

    5.4 Hive与Hadoop的关系

    SELECT rank
    ,isonline
    ,count(*)
FROM city GROUP BY rank,isonline
;

    处理过程:

    • 发出 SQL
    • Hive 处理，转换成 MapReduce
    • 提交任务到 Hadoop，MapReduce 在 HDFS 上运行

    5.5 Hive体系结构

    用户接口:

    • 用户接口主要有三个: CLI、Client 和 WUI。
    • 其中最常用的CLI，CLI 启动的时候，会同时启动一个 Hive 副本。
    • Client 是 Hive 的客户端，用户连接至 Hive Server。在启动 Client 模式的时候，需要指出Hive Server 所在节点，并且在该节点启动 Hive Server。
    • WUI 是通过浏览器访问 Hive。

    元数据存储:

    • Hive 将元数据存储在数据库中，如 mysql、derby
    • Hive 中的元数据包括表的名字，表的列和分区及其属性、表的属性(是否为外部表等)，表的数据素在目录等。

    经典数:

    解释器: 完成语法分析、语法分析，将HQL查询解析成AST

    逻辑计划生成器: 将 AST 转换为逻辑执行计划

    优化器: 对逻辑执行计划进行优化

    执行器:

    • 优化物理执行计划
    • 生成的查询计划存储在 HDFS 中，在 Hadoop 上面执行

    执行层: MapReduce 执行引擎

    存储层

    • Hive 的数据存储在 HDFS
    • 支持多种数据格式: 文本文件、RCFile 和 Parquet 格式等
    • 支持自定义数据格式解析器

    5.6 Hive数据存储

    采用关系型模型:

    • 数据库
    • 关系表

    5.7 物理存储

    依靠HDFS作为数据存储层: Hive 中包含以下数据模型: 表(Table)，外部表(External Table)，分区(Partition)，桶(Bucket)

    Hive的数据存储格式: 支持文本格式，Parqutet 列式存储格式，RCFile 以及 SequenceFile 等

    灵活简便的数据格式定义: 用户可以非常自由的组织 Hive 中的表，只需要在创建表的时候告诉 Hive 数据中的列分隔符和行分隔符，Hive 皆可以解析数据

    表:

    • Hive 中的表和关系型数据库中的表在概念上很类似
    • 每个表在 HDFS 中都有相应的目录用来存储表的数据，但数据是受 hive 管理的
    • 这个目录可以通过 ${HIVE_HOME}/conf/hive-site.xml 配置文件中的hive.metastore.warehouse.dir 属性来配置
    • 这个属性默认的值是 /uesr/hive/warehouse(HDFS上的目录)，我们可以根据实际的情况来修改这个配置

    外部表:

    • Hive 的外部表和表很类似
    • 但是其数据不是放在自己所属的目录中，而是存放到别处
    • 优势
      • 快速的数据导入，方便地集成外部数据资源
      • 如果你要删除这个外部表，该外部表所执行的数据是不会删除的，它只会删除外部表对应的元数据；而如果你要删除表，该表对应的所有数据包括元数据都会被删除

    分区:

    • 基于用户指定的分区列的值对数据表进行分区
    • 表的每一个分区对应表下的相应目录，所有分区的数据都是存储在对应的目录中
    • 这个分区知识逻辑上的分区

    分区的优点: 分区能用于查询的剪枝，提升查询的效率

    例: 分区示例

    CREATE TABLE orders_pbp(
    o_orderkey INT,
    o_orderstatus BIGINT,
    o_totalprice DOUBLE,
    o_orderdate DATE,
    o_orderpriority STRING,
    o_clerk STRING,
    o_shippriority INT,
    o_comment STRING
)
PARTITION BY (o_orderdate STRING) 
STORED AS parquet
;

    桶(bucket):

    • 基于用户指定列的值对数据表进行分桶
    • 对指定的类计算其 hash，根据 hash 值切分数据，目的是为了并行，每一个桶对应一个文件(注意和分区的区别)

    分桶的优点:

    • 获得更高的查询处理效率
    • 使取样(sampling)更高效

    例: 创建带桶的表

    CREATE TABLE orders_pbp(
    o_orderkey INT,
    o_orderstatus BIGINT,
    o_totalprice DOUBLE,
    o_orderdate DATE,
    o_orderpriority STRING,
    o_clerk STRING,
    o_shippriority INT,
    o_comment STRING
) PARTITION BY(o_orderdate STRING)
CLUSERED BY(o_orderkey) INTO 4 BUCKETS
SORTED BY(o_orderkey)
STORED AS parquet
;

    Map连接:

    • 对于 map 端连接的情况，两个表以相同方式划分桶。处理左边表内某个桶的 mapper 知道右边表内相匹配的行在对应的桶内。因此，mapper 只需要获取那个桶(这只是右边表内存储数据的一小部分)即可进行连接

    更高效的连接效率: 如果桶中的数据有序，对每个桶连接编程高效的归并排序(merge-sort)，因此可以进一步替身 map 端连接的效率

    更快速的采样: 可以基于某几个分区的数据进行采样

    • 数据倾斜: 数据分布不均匀

    • 倾斜的原因:

    6.3.2 Join数据倾斜

    • Map Join
    • 限制
      • 内存
      • 语义
    • 代价
    • 用法: select /* + MAPJOIN(tbl_alias) */
    • Bucketed map join
    • Sort merge bucketed map join

    6.3.3 GroupBy数据倾斜

    • skewindata 优化
    • 用法
      • set hive.groupby.skewindata=true: 数据倾斜时负载均衡
      • hive.map.aggr=true: Map端部分聚合，相当于 Combiner

    6.3.4 更好的执行计划

    • join 是一个使用频繁、同时又比较耗时的操作，使用时需要注意

    • 如果 join 时两个 table 都比较大，可以把较小的、或者 key 比较均匀的放在 join 左侧

    • 如果 join 时一个 table 大，一个 table 小到可以放入内存，可以使用 mapjoin，所有操作都会在 map 阶段完成

    • join 时可以把过滤条件放在 on 语句中，而不是 where 语句

    • 如果可以，推荐在一个 sql 语句中使用连续 join

    6.3.5 Hive合理控制Map和Reduce数

    1) Map数过大

    Map 阶段输出文件太小，产生大量小文件(下一个阶段就需要小文件合并，或者 reduce 阶段就会浪费 reduce 数)。初始化和创建 map 的开销很大

    2) Map数过小

    • 文件处理或查询并发度小，Job 执行空间过长

    • 大量作业时，容易堵塞集群

    减少map数和增加map数

    ※ 减少 map 数可以通过合并小文件来实现，这点是对文件源。

    set hive.merge.mapredfiles=true;， 默认 false，true时在MapReduce的任务结束时合并小文件

    ※ 增加 map 数的可以通过控制上一个 job 的 reduce 数来控制(一个 sql 中 join 多个表会分解为多个 mapreduce)

    Hive中设置 map 数: 参数 mapred.map.tasks。如 set mapred.map.tasks=100;，并不是每次都是有效的，要看大小是否合理

    3) Reduce数过大

    生成了很多个小文件(最终输出文件由 reduce 决定，一个 reduce 一个文件)，那么如果小文件作为下一个 Job 输入，则也会出现小文件过多需要进行合并（耗费资源）的问题。

    启动和初始化 reduce 也会消耗大量的时间和资源，有多少个 reduce 就会有多少个输出文件

    4) Reduce过小

    • 每个文件很大，执行耗时
    • 可能出现数据倾斜

    5) reduce个数的决定

    默认下，Hive 分配 reduce 数基于以下参数:

    • 参数1: hive.exec.reducers.bytes.per.reducer (默认是 1G)

    • 参数2: hive.exec.reducers.max (最大 reducer 数，默认是 999)

    6) 计算reducer数的公式

    N=min(参数2，总输入数据量 / 参数1)，即默认一个 reduce 处理 1G 数据量。

    7) 会话级别参数调优

    数据量小于 hive.exec.reducers.bytes.per.reducer 参数值，或者没有 group by，或者使用了 order by 语句造成数据倾斜，可以通过会话级别参数调优

    8) 解决

    设置 reduce 数，参数 mapred.reduce.tasks 默认是 1。如，set mapred.reduce.tasks=10;

    7. Zookeeper的原理安装及应用

    7.1 Zookeeper的原理和部署

    Zookeeper 是一个分布式的、开放源码的分布式应用程序协调服务，是 Google 的 Chubby 一个开源的实现。

    它是集群的管理者，监视着集群中各个节点的状态根据节点提交的反馈进行下一步合理操作。

    最终，将简单易用的接口和性能高效、功能稳定的系统提供给用户。

    Zookeeper 的核心是原子广播，这个机制保证了各个 server 之间的同步。实现这个机制的协议叫做 Zab 协议。

    Zab 协议有两种模式，分别是恢复模式(选主)和广播模式(同步)。

    当服务启动或者在领导者崩溃后，Zab 就进入了恢复模式；当领导者被选举出来，且大多数Server 完成了和 Leader 的状态同步以后，恢复模式就结束了。

    状态同步保证了 Leader 和 Server 具有相同的系统状态。

    为了保证事务的顺序一致性，zookeeper 采用递增的事务 id 号(zxid) 来标识事务。所有的提议(proposal) 都在被提出的时候加上了 zxid。

    现实中 zxid 是一个 64 位的数字，它高 32 位是 epoch 用来标识 Leader 关系是否改变，每次一个 Leader 被选出来，它都会有一个新的 epoch，标识当前属于哪个 leader 的统治时期。低 32 位用于递增计数。

    7.2 Zookeeper的应用场景

    7.2.1 Zookeeper在Hadoop平台典型应用

    Storm集群:

    Zookeeper 作为 nimbus(master) 和 supervisor(slave) 的中间枢纽额，保存 Storm 集群和作业的所有信息，并负责 nimbus 和 supervisor 的全部通信，Fast fail。

    HBase集群:

    Zookeeper 作为”协调器”，为 HBase 提供了稳定服务和 fail over 机制。HRegionServer 也会把自己以 Ephemeral 方式注册到 Zookeeper 中，使得 HMaster 可以随时感知到各个HRegionServer 的健康状态(可用于监控 RegionServer)。此外，Zookeeper 也避免了HMaster的单点问题。

    应用三大块:

    • Storm应用开发，Storm集群监控

    • MapReduce应用开发

    • HBase应用开发

    7.2.2 Zookeeper的配置管理

    zoo.cfg 是 zookeeper 的配置文件

    参数说明:

    • tickTime : 集群之间间隔心跳的时间，单位为ms。

    • initLimit : 初始化花费的时间(心跳的次数)

    • syncLimit : 同步花费的时间(心跳的次数)

    • dataDir : 为 zookeeper 下为集群数据保持一直最重要的一点

    • dataLogDir : 是 zookeeper 的日志文件产生目录

    注: server.1, server.2, server.3 的端口号不一定得一致

    程序总是需要配置的，如果程序分散部署在多台机器上，要逐个改变配置就变得困难。现在把这些配置全部放到 zookeeper 上去，保存在 Zookeeper 的某个目录节点中，然后所有相关应用程序对这个目录节点进行监听，一旦配置信息发生变化，每个应用程序就会收到 Zookeeper 的通知，然后从 Zookeeper 获取新的配置信息应用到系统中就好。

    Hadoop的HA机制: 监控主要是 3 部分，分别为数据、配置信息、名称管理。

    8. HBase的原理及应用场景

    8.1 HBase的基本原理

    8.1.1 HBase-Hadoop Database

    • Google 的 BigTable 系统的开源实现
    • HBase 利用 Hadoop HDFS 作为其文件存储系统
    • 利用 Hadoop MapReduce 来处理 HBase 中海量数据
    • 利用 Zookeeper 作为协调工具，用于管理集群

    8.1.2 为何使用HBase？

    数据分类与存储: 结构化，非结构化和无结构化

    NoSQL(Not Only SQL)

    • High performance : 对数据库的高并发读写
    • Huge Storage : 对海量数据的高效率存储和访问需求
    • High Scalability and High availability : 7*24 的在线服务，数据库通过增加机器来进行横向扩展

    8.1.3 HBase VS. RDBMS(关系型数据库)

    常用的非关系型数据库: Mongodb、Redis、InfoGrid、HBase

    RDMS:

    • 支持关系模型
    • 支持事务的 ACID 原则
    • 提供 SQL 查询接口

    HBase:

    • BigTable 数据模型：稀疏的宽表模型，每行的列数不固定
    • 仅支持单行操作上的事务
    • 提供 API 以及命令行命令
    • 分布式列存储数据库

    注: CS(Client-Server)

    8.1.4 HBase的逻辑数据模型-行

    • 表内数据是由行组成的，不同的行的列的数完全可以大不相同
    • 数据烈性: 字符串、整数、二进制甚至串行话的结构
    • 表按照行键的逐字节排序顺序对行进行排序的
    • 可以只对一行上锁
    • 对行的写操作是始终是原子的

    8.1.5 Hbase的逻辑数据模型-列簇

    • Table 在水平方向由一个或者多个 Column Family 组成
    • 一个 Column Family 中可以有任意多个Column 组成，即 Column Family 支持动态扩展
    • 无需预先定义 Column 的数量以及类型，所有 Column 均以二进制格式存储，用户需要自行进行类型转换
    • 每个 Column 可以有多个版本，通过 timestamp 标识

    8.1.6 HBase物理存储-表

    • Table 在行的方向上分割为多个 HRegion
    • region 按大小分割的，每个表开始只有一个 region，随着数据不断插入表，region 不断增大，当增大到一个阈值的时候，Hregion 就会等分会两个新的 Hregion。当 table 中的行不断增多，就会有越来越多的 Hregion。
    • HRegion大小的设置: 参数 hbase.hregion.max.filesize

    8.1.7 Hbase物理存储-HRegion

    • HRegion 是 Hbase 中分布式存储和负载均衡的最小单元
    • 最小单元就表示不同的 Hregion 可以分布在不同的 HRegion server 上。
    • 但一个 Hregion 是不会拆分到多个 server 上的

    • HRegion 虽然是分布式存储的最小单元，但并不是存储的最小单元
    • Region 由一个或者多个 Store 组成，每个 store 储存一个 columns family
    • 每个 Store 又由一个 memStore 和像 0 至多个 StoreFile 组成
    • StoreFile 以 HFile 格式保存在 HDFS 上

    8.1.8 Hbase物理存储-HFile

    • Data Block 段: 保存表中的数据，这部分可以被压缩
    • Meta Block段(可选的): 保存用户自定义的 key、value 对，可以被压缩
    • File Info 段: Hfile 的元信息，不被压缩，用户也可以在这一部分添加自己的元信息
    • Data Block Index 段: Data Block 的索引。每条索引的 key 的被索引的 block 的第一条记录的 key
    • Meta Block Index 段(可选的) : Meta Block 的索引
    • Trainer: 这一段是定长的。保存了每一段的偏移量，读取一个 HFile 时，会首先读取Trainer。

    8.1.9 Hbase体系架构

    • Master server
      • 为 Region server 分配 region
      • 负责 Region server 的负载均衡
      • 发现失效的 Region server 并重新分配骑上的 region
      • 管理用户对 table 的增删改查操作
    • Region server
      • 维护 Master 分配给它的 region，处理对这些 region 的 IO 请求
      • 负责切分在运行过程中变得过大的 region
    • Zookeeper
      • 集群管理
      • 元数据管理
    • HDFS
      • 包含访问 Hbase 的接口
      • 并维护 cache 来加快对 HBase 的访问，比如 region 的位置信息

    可以看出，client 访问 hbase 上数据的过程并不需要 master 参与，寻址访问 zookeeper 和region server，数据读写访问 region server。

    HRegion Server 主要负责相应用户 IO 请求，想 HDFS 文件系统中读写数据，是 HBase 中最核心的模块。

    static class BatchImportMapper extends Mapper<LongWritable, Text, LongWritable, Text>{
     SimpleDateFormat dateformat1 = new SimpleDateFormat("yyyyMMddHHmmss");
     Text v2 = new Text();

     protected void map(LongWritable key, Text value, Context context) throws java.io.IOException ,InterruptedException {
     final String[] splited = value.toString().split("\t");
     try {
          final Date date = new Date(Long.parseLong(splited[0].trim()));
          final String dateFormat = dateformat1.format(date);
          String rowKey = splited[1] + ":" + dateFormat;
          v2.set(rowKey + "\t" + value.toString());
          context.write(key, v2);
     } catch (NumberFormatException e) {
          final Counter counter = context.getCounter("BatchImport", "ErrorFormat");
          counter.increment(1L);
          System.out.println("出错了" + splited[0] + " " + e.getMessage());
          }
     };
}

    8.3 Hbase应用场景

    1) 物理架构

    在 Hadoop 的各层系统，Hbase 位于结构化存储层，hadoop HDFS 为 HBase 提供了高可靠性的底层存储支持，Hadoop MapReduce 为 HBase 提供了高性能的计算能力，Zookeeper 为 HBase 提供了稳定服务和管理。

    2) HBase的内部主要分为三部分

    客户端Client，HMaster进程，HRegionServer进行管理

    • HMaster 通常是主备模式，由 Zookeeper 选取，对多个 HRegion Server 进行管理
    • 一个 HRegion Server 对应着一个承载数据的Region好二级路日志 HLog

    3) HBase适合具有以下需求的应用

    • 海量数据(TB, PB)
    • 高吞吐量，实时性
    • 能够在海量数据中实现高效的读取
    • 很好的延展性，动态扩展整个存储系统容量
    • 能够同时处理结构和非结构化的数据

    4) HBase 是一个分布式的存储系统，可以在廉价的 PC 上搭建大规模存储系统，用于存储海量数据，适合于作为站点数据统计工具的存储系统

    5) 对实时数据的统计，hbase 能够提供较低延迟的读写访问，承受高并发的访问请求；对历史数据的统计，Hbase 可以视为一个巨大的 key-value 存储系统，由于存储各个网站上历史的访问信息

    6、对于像 PV、UV、IP 这样需要求累加计算的操作(求 SUM/AVG)，由于数据量巨大，关系型数据库保证不了响应速度，而是用 HBase 可以对服务端进行聚合计算，这样就请求 HBase 响应大量数据进行传输，直接请求服务端(集群) 计算的结果，因此能够满足实时响应的需求。

    9.1 Hbase的安装配置说明

    /bin : binary文件都是可执行文件，.sh、.cmd、.exe、.bat

    /lib : 都是依赖的组件包 *.jar

    hadoop生态系统中的配置文件类型:

    • .xml 配置文件。比较通用的，用任何一个高级语言都可以解析的文件类型

    • .property 或者 properties : 属性文件，使用 java 特有格式形成的一个文件类型

    • Hbase 里面文件类型是 HFile，HFile 和 HDFS 文件不是同一个类型，HFile 是 HDFS 文件的一个种类

    • hive 里面知识保存数据的结构(描述)，数据本身是放在 HDFS 里面的

    • HBase 的运行依赖 zk，hbase 在安装的时候，里面有一个自带的 zk，同时也可以不使用自带的，而是用另外单独的 zk(建议另外配置zk)

    • xml 和 sh文件，这些配置项是 hbase 启动的时候所必须依赖的，并不是每个配置项都必须在启动前配置好。

    • 在 hbase 启动的时候，入伙配置项没有设定在文件中，就使用代码中的默认值

    • 可以设定 xml 或 sh 文件，来对一些核心内容进行手动配置

    • 还可以在 java 或 python 程序里面，动态的设定某些配置项的值

    hbase.rootdir 这个目录是 region server 的共享目录，用来持久化 Hbase。URL 需要是 '完全正确的'，还包含文件系统的 scheme。

    例如: 要表示 hdfs 中的 '/hbase' 目录，namenode 运行在 namenode.example.org 的 9090 端口。则需要设置为 hdfs://namenode.example.org:9090/hbase。

    默认情况下 Hbase 是写到 /tmp 的。不改这个配置，数据会在重启的时候丢失。

    默认是 file:///tmp/habse-${user.name}/hbase (本地文件系统)

    9.3 Hbase的性能调优

    1) 传统行式数据库

    • 数据是按行存储的
    • 没有索引的查询使用大量 I/O
    • 建立索引和物化视图需要花费大量时间和资源
    • 面对查询的需求，数据库必须被大量膨胀才能满足性能要求

    2) 列式数据库

    • 数据按列 - 每一列单独存放
    • 数据即是索引
    • 只访问查询涉及的列 —— 大量降低系统 IO
    • 每一列由一个线索来处理 - 查询的并发处理
    • 数据类型一致，数据特征相似 —— 高效压缩

    3) swap的设置

    我们需要关闭系统的 swap 或是这只 swapiness 为 0，推荐设置为 0，在物理内存不够的情况下才会使用交换分区。这个参数设置是由于 JVM 虚拟机如果使用了 swap 在 GC 回收时会花费更多的时间，会导致 Region server 与 ZK 连接超期，Hmaster 会认为 Region server 已经故障，然后开始分裂 HLOG 和重新分配 Region，及时 Region server 完成 GC 后，再次上报信息给 Hmaster 时，Hmaster 也会抛 YouAreDead 的异常给 Region server 和让它退出服务，所以这个参数的配置是一个必须条件。

    4) GC 回收采用并 hang 增量式的方式，要设置 ZK 的 session 时长和内存大小到一个比较合适的值，减少因为 full GC 产生宕机的情况。另外，我们一定要注意一下参数

    --XX:CMSInititatingOccupancyFraction=70

    70 为 JVM 的使用百分比，当达到这个阈值后将启动回收任务。这个值比较合适的值要略大于 memstoresize 40% + blockcache 20%

    5) zookeeper.session.timeout(默认3分钟)

    ZK 的超期参数，默认配置为 3 分钟，在生产环境上建议减少这个值在 1 分钟或更小

    设置原则: 这个值越小，当 RS 故障时 Hmaster 获知越快，Hlog 分裂和 region 部署越快，集群恢复时间越快。但是，设置这个值得原则是留足够的时间进行 GC 回收，否则会导致频繁的 RS 宕机

    6) hbase.regionserver.handler.count(默认是10)

    设置原则：

    • 对于大负载的 put(达到了 M 范围) 或是大范围的 Scan 操作，handler 数目不宜过大，易造成 OOM

    • 对于小负载的 put 或是 get、delete 等操作，handler 数要适当调大。

    根据上面的情况，要看我们的业务的情况来设置

    7) HBASE_HEAPSIZE 大小设置

    设置原则：

    • HBASE_HEAPSIZE 包括三部分内容: Memstoresize 40%(默认)、blocksize 20% 以及StoreIndex (这需要根据硬盘容量算出)
    • GC 回收的时间不要超过 zk session 的时间

    8) 表的设计

    ROWKEY 设计要离散度好，不要在一张表里定义太多的 columns family，将表放到 Region Server 的缓存中

    9) 写表操作

    创建多个 HTable 客户端用于写操作，提高写数据的吞吐量，通过调用HTable.setAutoFlush(False) 方法可以将 HTable 写客户端的自动 flush 关闭，这样可以批量写入数据到 Hbase

    10) 读表操作

    创建多个 HTable 客户端用于读操作，提高读数据的吞吐量，对于频繁查询 HBase 的应用场景，可以考虑在应用程序中做缓存，当有新的查询请求时，首先在缓存中查找，如果存在则直接返回，不再查询 Hbase；否则对 Hbase 发起读请求查询，然后在应用程序中将查询结果的缓存起来

    11) 计算操作

    Hbase 本身可以看做是一个可以水平扩展的 key-value 存储系统，但是其本身的计算能力有限(Coprocessor可以提供一定的服务端计算)，因此，使用 Hbase 时，往往需要从写端或者读端进行计算，然后将最终的计算结果返回给调用者。
