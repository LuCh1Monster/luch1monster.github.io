    1. 更新 Hive 元数据到 Impala

    有时候数据在 Hive 已经更新了，但是 Impala 中却始终获取不到数据，原因在于元数据在 Impala 中没有刷新。需要在 Impala 中执行如下代码来更新数据(在 Hive 中是不可以执行的，且执行是无效的):

    invalidate metadata [db.tb_name]
;

    2. 创建 Impala 外部表

    1. 上传 CSV 文件至 HDFS。注意: CSV 文件的列名应该去掉，否则创建出来的表第一行会存在问题。
    2. 如果是分区表，表名对应一个文件夹，每个分区应为该文件夹下的一个子文件夹，且文件夹中存放的数据为分区数据。如果不是分区表，只是一张外部表，则只需要创建一个文件夹对应该表的数据。
    3. 在 Impala 或者 Hive 的编辑器中执行下面代码。
    --外部表
create external table test.table1(
    字段 类型,
    ...
)
row format delimited fields terminated by ',' 
stored as textfile
location '/PATH/to/tb_directory'

--外部分区表
create external table test.table2(
    字段 类型,
    ...
)
PARTITIONED BY ( `dt` string)
row format delimited fields terminated by ','
stored as textfile location '/PATH/to/tb_directory'
;

    3. 加载分区

    我们刚刚创建了一张分区表 test.sales ，对应的 HDFS 目录为 /webhdfs/yumingmin/sales，现在需要给这张表添加 dt 分区数据。首先需要你要确保该目录下存在 dt 的目录，应该是每天一个 dt 文件。例如，我们需要添加 2019-01-01 数据分区，首先确认 dt=2019-01-01 的文件夹存在，且该文件夹存放着 CSV 文件。关于 CSV 文件的格式，可以参考是上面的格式。

    现在可以在 Hive 或者 Impala 中执行以下 SQL 语句，来挂载 2019-01-01 的分区数据:

    alter table test.sales drop if exists partition(dt="2019-01-01")
;

alter table test.sales add partition(dt="2019-12-19") location "/webhdfs/yumingmin/sales/dt=2019-12-19"
;

    这样就将分区数据挂载好了，如果你在 Hive 执行的上面代码，还需要使用 invalidate metadata test.sales 或者 refresh test.sales 来刷新数据。

    4. refresh 和 invalidate metadata 的区别

    这两个操作都是用来刷新元数据，一般有两种情况需要刷新源数据:

    • 元数据过期
    • 元数据有缺失

    区别 : 这两个命令都得在 Impala 中使用，Hive 中是不可使用的。

    • refresh 是一个轻量级的操作，适用于表结构没有发生改变时使用。

    • invalidate metadata 是当表结构发生了改变后来进行元数据的刷新。

    5. 外部表和内部表的区别

    • 外部表删除后，数据仍然存在

    • 内部表删除后，数据也会删除

    • 对于外部表，需要挂载分区后才能看到数据。

    6. 表的基本格式

    一般创建表时，有 2 种格式: textfile 和 parquet。如果指定格式，默认使用 stored as textfile 格式进行表数据的存储。非特殊情况下，还是使用 parquet 最好。

    注意 : 在 parquet 格式下，to_date 转换的字段可能无法存储，导致报错，这取决于 Impala 的版本。这种情况下，需要将字段再 cast 成 string 类型。

    • show table stats :
    • show columns stats :
    • show partitions :
    • describe formatted : 比 describe 查看的信息更全，可以看到表的 Owner、最后一次更新时间等等。

    8. Join策略

    Join 的策略分为 broadcast 和 partitioned。

    8.1 broadcast join

    broadcast join 主要用于小表的 Join。

    8.2 partitioned join

    partitioned join 使用的标志是 [shuffle]，它主要用于大表的 Join。

    8.3 broadcast 和 partitioned join 的区别

    当表缺少统计信息的时候，Impala 默认执行策略是 broadcast join。Broadcast 是将执行计划中join 右侧的表广播到执行 sql 的各个节点。这是 sql 慢的主要原因之一，这种情况会严重占用集群带宽，造成其他简单 sql 也运行缓慢！

    9. SQL优化

    9.1 Explain 查看执行计划

    执行计划会告诉我们:

    • SQL 预计需要占用的内存
    • SQL 所用的表是否缺少统计信息
    • SQL 所扫描的数据量，如果是分区表，会看到使用了多少个分区
    • SQL Join 的策略
    • …

    当我们的 SQL 运行了超过 30 分钟，还没有出结果，此时需要去使用 explain 来查看执行计划。

    9.2 如何调优

    1) 收集统计信息

    • 非分区表: compute stats
    • 分区表: compute incremental stats，对于指定的分区，可以使用 compute incremental stats [table] partiton(dt="2019-01-01")

    2) 使用分区字段来缩小查询范围，具体可以在 where 和 join 中进行使用

    3) straight join

    • 自动调优规则：join query 中部分表缺少统计信息时，有统计信息的表会放在 join 的左侧，根据总体大小和基数按成本降序排列，没有统计信息的表放在 join 的右侧。
    • Impala 忽略自动调优，按照 SQL 连接顺序去执行，写在 select 或 distinct 或者其他关键字后
    • 对内层的子句(嵌套、where 子句)无作用

    4) [shuffle]

    • 添加在 join 后，将 broadcast 转换成 partitioned
    • 再次 explain 查看执行计划，确认是否优化成功
    • 外层 shuffle，对内层的 join 子句是不起作用的

    5) limit

    • 大表仅筛选几条的情况下使用，否则很容易出现“数据加载中”，进而导致下一条 SQL 无法提交
    • 结果只有2行，结果越少越要加!

    6) 将频繁使用到的逻辑固化成为特定的中间表，减少查询消耗

    7) 避免使用 SELECT *

    10. INSERT INTO 和 INSERT OVERWRITE

    INSERT [INTO|OVERWRITE] TABLE tb1
SELECT * FROM tb2;

INSERT [INTO|OVERWRITE] TABLE tb1 VALUES ...;

insert into t1 partition (dt='2019-04-18') select c1 from t2;

insert into t1 partition (dt) select c1, dt from t2;

    当需要重新表的全部数据，一种方法是先 DROP 再 CREATE，还有一种方法是 INSERT OVERWRITE，后者更高效、稳定。

    INSERT VALUES 应该尽量规避使用，会产生小文件，应该用 INSERT SELECT 代替。

    11. LOAD DATA

    在 Terminal 中可以运行以下数据来更新表数据:

    load data local inpath "/home/hadoop/bj.txt" into table stud;

-- 分区表
load data local inpath "/home/hadoop/bj.txt" into table stud_ptn partition(city="bj")

    12. 常用函数

    • nvl :
      • nvl(expr1, expr2)测试表达式是否空值。如果是空值，返回 expr2 结果；如果不是空值，则返回表达式 expr1 结果值
      • nvl(expr1, expr2, expr3)测试表达式是否空值。如果是空值，返回 expr2 结果；如果不是空值，则返回表达式 expr3 结果值
    • substr、strleft、strright
    • dround : 四舍五入
    • grest/least : 取最大最小值
    • concat/concat_ws : 字符串的拼接
    • add_months(now(), 1) : 当前时间增加一个月
    • adddate(now(), 1) : 当前日期加一天，等效于 date_add(now(), 1)
    • datediff(dt1, dt2): 计算两日期天差
    • date_sub : 日期减函数
    • unix_timestamp : 当前日期距离1970年的秒数
    • from_unixtime(1556108753,“yyyy-MM-dd”) : 秒数转化为日期
    • year、month、day、hour、minute、second
    • current_timestamp()、now()

    ALTER TABLE NEW_STUDENT REPLACE COLUMNS (ID INT, NAME STRING, ADDRESS STRING);

    14.6 删除表

    DROP TABLE tb;

TRUNCATE TABLE tb;

    15. Hive与Impala异同点

    15.1 相同点

    • 数据存储 : 使用相同的存储数据池都支持把数据存储于HDFS, HBase。
    • 元数据 : 两者使用相同的元数据。
    • SQL解释处理 : 比较相似都是通过词法分析生成执行计划。

    15.2 不同点

    • 执行计划:
      • Hive : 依赖于 MapReduce 执行框架，执行计划分成 map -> shuffle -> reduce -> map -> shuffle -> reduce… 的模型。
      • Impala : 把执行计划表现为一棵完整的执行计划树，可以更自然地分发执行计划到各个 Impalad 执行查询
    • 数据流:
      • Hive : 采用推的方式，每一个计算节点计算完成后将数据主动推给后续节点。
      • Impala : 采用拉的方式，后续节点通过 getNext 主动向前面节点要数据，数据可以流式的返回给客户端
    • 容错:
      • Hive：依赖 Hadoop 的容错能力
      • Impala：没有容错逻辑
    • 适用面:
      • Hive : 复杂的批处理查询任务，数据转换任务
      • Impala : 实时数据分析
    • 函数用法：
      Hive : 全面，符合 SQL 标准
    • Impala : Hive中某些函数不支持，比如 count(distinct column) 不能超过一个

    16. 相关概念

    16.1 数据倾斜

    • group by 字段上的数据分布不均匀
    • Join 字段上的数据分布不均匀
    • Left join 以后用有大量空值的字段再做 join

    17. WITH

    • 同一段 with 子句多次引用会多次执行
    • 对于会重复引用的 with 子句，建临时表
    • WITH 应该尽量少用
